# Refactor: typed jj facade → jj-lib in-process (2026-07-16)

The plan-and-design document for the jj refactor program.
Version-control operations are scattered hand-rolled
`run("jj", …)` spawns (~30 query sites plus every mutation)
with per-module private wrappers, plus raw-git vestiges in
init. Spawning means stderr parsing instead of typed errors,
and jj's single-attempt index-lock acquisition sits outside
our control — the push `bookmark-set` lock race [[1]] can't
be retried where it fails.

How this file works:

- Each stage below is a `##` section in execution order;
  each is roughly one `X.Y.Z` cycle. Section anchors are the
  durable way to reference a stage from TODO.md, chores, or
  bugs.md.
- This file owns the forward plan and design. When a stage
  ships, its `chores-NN.md` section records the commits and
  narrative as usual; the stage section here gets a one-line
  status link to that chores section — no duplication.
- The `## Todo` entry "Refactor: typed jj facade → jj-lib
  in-process; end subprocess spawning" in
  [TODO.md](../TODO.md) is the short pointer here.

Absorbed former Todos: "Dedup jj subprocess queries behind a
typed facade", "Split push.rs state machinery", "vc-x1 push:
validate body opens with an intro paragraph", "Make the bot
repo directory name configurable", "Terminology stragglers
from the work/bot sweep", "single-field `options_flags`
leaves → `value` field", "`por → dual` conversion",
"`validate-desc` / `fix-desc` por equalization".

## Stage: DRY facade

~30 call sites hand-roll `run("jj", ["log", "-r", <rev>,
"--no-graph", "-T", <template>, "-R", <repo>])` and each
module has quietly grown a private wrapper. Spotted at
0.69.0-2: `jj_rev_exists` read as "first of its kind" but is
the Nth reinvention.

- One facade module (name open: `src/jj.rs` or a `common`
  split): `jj_log(repo, rev, template)` plus typed helpers —
  `rev_exists`, `chid_of`, `cid_of`, `desc_of`, `is_empty`.
- Existing reinventions to fold:
  `squash_push::{jj_rev_exists, jj_commit_id,
  rev_is_empty_undescribed}`, `push::jj_log_empty` + four
  inline `change_id.short(12)` blocks, `init::jj_chid`,
  three template variants in `sync.rs`.
- Fold `main::bm_track_one` (raw `std::process::Command` +
  its own `@origin:` prefix parse) onto
  `common::verify_tracking`'s parser — two tracking parsers
  will eventually disagree.
- Unify string-level ochid trailer parsing:
  `squash_push::extract_ochids` vs `common::extract_ochid`
  (jj-lib `Commit`-based); one string-level parser in
  `desc_helpers` both can call.
- Test dedup: promote the near-identical `jj()` / `cid()` /
  `chid()` / `description()` helpers from
  `push/integration_tests.rs`, `sync/integration_tests.rs`,
  and `tests/cli_sync.rs` into `test_helpers.rs`.

## Stage: hygiene riders

Sweeps that churn the same lines the facade stage rewrites;
done early so later stages diff cleanly.

- Terminology stragglers from the 0.69.0-4 work/bot sweep:
  - `-s` scope keyword `code` (renaming to `work` is a
    breaking CLI change; `bot` already matches)
  - init's literal remote names `remote-code.git` /
    `remote-claude.git`
  - identifiers `derive_session_url`, `claude_path`,
    `Fixture.claude`, `Side::Code`
  - decide rename vs document-as-historical (overlaps
    [Stage: facade owns topology](#stage-facade-owns-topology))
- Single-field `options_flags` leaves → `value` field (the
  0.47.0 convention from the `squash` leaf): sweep `repo`,
  `dry_run`, `private`, `account`, `config`, `use_template`
  + consumers (`init.rs`, tests). Open: could a single field
  be a type/enum instead of a struct, eliminating the
  `args.<leaf>.<leaf>` naming issue entirely.

## Stage: facade owns topology

Repo resolution becomes facade state, not per-command
preludes.

- `validate-desc` / `fix-desc` por equalization: replace the
  `other_repo_from_config` prelude in both
  (`validate_desc.rs:133`, `fix_desc.rs:152`) with
  scope-aware resolution that no-ops `Side::Bot` when
  absent; body unchanged. The 0.61.0 audit [[2]] called this
  the cheapest equalization and the prototype for the
  topology-from-config rule (topology from `default_scope`,
  not a flag); the remaining implementation gaps live in
  [[2]]'s `## Gap list`.
- Make the bot repo directory name configurable (not
  hardcoded `.claude`; decided 2026-07-15 the name need not
  be `.claude`). The literal is embedded across
  init/clone/push/sync, `claude_path()`, ochid path
  semantics (`/.claude/<chid>`), the
  `~/.claude/projects/<path>` symlink, and docs;
  `.vc-config.toml` already records workspace paths — the
  natural home. Open: migration of existing workspaces, the
  ochid prefix recorded in immutable history, the
  harness-controlled symlink location.

## Stage: de-gitify init

`init.rs::push_repo`'s strip-jj → git-push → re-colocate
flow is the step shape the abandoned submodule design
required [[3]]; the symlink design doesn't need jj evicted.

- jj-only replacement verified on jj 0.40.0 (2026-07-16,
  scratch run): `jj git init --colocate <dest>` (creates the
  git repo itself — the explicit `git init` in
  `prepare_local_repo` is redundant), `jj commit`,
  `jj git remote add origin <url>`, `jj bookmark set main
  -r @-`, `jj git push --bookmark main`.
- No `--allow-new` needed; push established tracking as a
  side effect (matches the test-fixture observation in
  [[4]]).
- Collapses Step 7/Step 10; drops `git clean` /
  `git checkout` / `git remote add` / `git push -u`.

## Stage: split push.rs

`push.rs` (~1.6k lines, the largest file) holds the `Stage`
machine, TOML state persistence, eight stage bodies, two
sanity verifiers, and the interactive gates; `sync/state.rs`
is the in-repo precedent.

- Extract `push/state.rs` (`Stage`, `PushState`, state
  layout resolution) so the
  [jj-lib migration](#stage-jj-lib-migration) rewrites stage
  bodies in a file that reviews cleanly.

## Stage: jj-lib migration

Migrate the facade internals to jj-lib (already a dependency
for read-only commands); removes jj and git spawning
entirely.

- Reads first: `jj log` templates become `Commit` accessors.
- Then mutations: commit, describe, bookmark set/track,
  fetch, push — the workspace/transaction/op-store lift.
- The index-lock retry loop becomes ours (catch, back off,
  retry) with real error types — the fix vehicle for the
  bugs.md `bookmark-set` lock race [[1]].
- `LocalBareInit`'s `git init --bare` becomes a gix call
  (jj-lib already pulls gix in).

## Stage: push body-intro validation

A bullet-first commit body trips jj's arg parser today
(`-m "<body>"` reads the leading `-` as a stray flag; hit on
0.62.0-5) and push fails with an opaque error.

- Validate the body opens with a non-dash intro line (Prose
  Form) and report the offending first line with a clear,
  specific error — enforcing the intro is the intended
  behavior, not a parser fix.
- Workaround until then: prepend a non-dash intro sentence.

## Stage: trapezoid close-out

The close-out commit becomes a two-parent merge — first
parent the pre-cycle base (the parent of the ladder's first
commit), second parent the ladder tip (`-X`), the
`git merge --no-ff` convention — so
`git log --first-parent main` reads one commit per cycle
while the full ladder stays reachable via the second parent.

- The merge itself is trivial in-process: a two-parent
  commit in one transaction; no conflicts possible since the
  base is an ancestor of the tip.
- The real work is ochid stamping — the bot commit carries
  one ochid per work commit in the cycle (the AGENTS.md
  merge non-ff shape) while push today stamps only the
  topmost commit.
- Teach `validate-desc` + the push sanity verifiers the new
  legal shape (two parents, N ochids).
- End state of the Todo "vc-x1 push: pause point between
  commit and publish stages", which stays as the interim
  manual recipe until this lands.

## Stage: por → dual conversion

Attach a bot companion repo + `.vc-config.toml` to an
existing por workspace; emit cross-links going forward.
Manual setup on an external por workspace (2026-05-14)
proved arduous; should be a routine subcommand. Design stub
in [[5]] § 2.

- Last on purpose: it leans on
  [facade topology](#stage-facade-owns-topology) and the
  in-process init pieces.

## Access rule

Today's implicit rule — read-only display commands use
jj-lib in-process, mutating commands shell out — dissolves
once mutations are in-process; ARCHITECTURE.md states the
end state instead.

## Out of scope: still spawned

- `gh` (remote provisioning) and `$EDITOR` stay spawned by
  design.
- The `vc-x1 sync --check` self-spawn goes away when the
  Todo "sync follow-up: push preflight in-process" lands —
  same direction, independently shippable, not part of this
  program.
- Beneficiary: the Todo "Run validate-bot at every vc-x1
  invocation" — its per-invocation jj spawn becomes an
  in-process read.

## Design risk: op-store coexistence

We think in-process op-store writes must coexist with the
user's external `jj` CLI driving the same repos; jj-lib 0.41
vs installed jj 0.40.0 format/version skew needs a
compatibility check before the
[mutation stage](#stage-jj-lib-migration) commits to the
approach.
